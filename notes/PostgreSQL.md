# 1. 使用PG咨询锁替代Redis分布式锁

在 PostgreSQL（PG）中替代 Redis 实现分布式锁，最正统且高效的方式是利用其内置的**咨询锁（Advisory Locks）**。这种锁与数据库的表结构或行数据解耦，专门用于应用程序级别的并发控制。

## 1.1. 核心方案：PostgreSQL 咨询锁 (Advisory Locks)

PG 咨询锁允许你通过一个 64 位整数（或两个 32 位整数）作为锁的标识符。它不需要像 Redis 那样手动管理过期时间（TTL），因为它可以与数据库会话（Session）绑定。

A. 会话级锁 (Session-level)

锁与当前的数据库连接绑定。如果连接断开（如应用崩溃），PG 会**自动释放**该锁，天然避免了 Redis 中常见的死锁问题。

- **加锁**：`SELECT pg_advisory_lock(锁ID);`（阻塞直到获取）或 `SELECT pg_try_advisory_lock(锁ID);`（非阻塞，返回布尔值）。
- **解锁**：`SELECT pg_advisory_unlock(锁ID);`。

B. 事务级锁 (Transaction-level)

锁在事务结束（Commit 或 Rollback）时自动释放。

- **加锁**：`SELECT pg_advisory_xact_lock(锁ID);`。
- **释放**：无需手动释放，事务结束即消失。

------

## 1.2. PG 替代 Redis 的实现对比

| 特性           | Redis 实现 (SETNX + Lua)                                 | PostgreSQL 实现 (Advisory Lock)                  |
| :------------- | :------------------------------------------------------- | :----------------------------------------------- |
| **可靠性**     | 依赖过期时间防止死锁；极端情况下（如 Redlock）实现复杂。 | **高**。利用数据库连接状态，会话断开即自动释放。 |
| **复杂度**     | 需处理原子性、续期（Watchdog）和过期释放。               | **低**。内置函数，事务支持，语义清晰。           |
| **性能**       | 极高（内存操作）。                                       | 中高（磁盘 IO 较少，主要受限于连接池开销）。     |
| **数据一致性** | 最终一致性（异步复制）。                                 | **强一致性**。基于 WAL 日志和数据库 ACID 特性。  |

------

## 1.3. 具体操作示例

如果你想在 Java、Python 或 Go 等后端代码中使用，可以直接执行 SQL：

1.3.1. **非阻塞获取锁**：

   ```sql
   -- 尝试获取名为 12345 的锁，成功返回 true，失败立刻返回 false
   SELECT pg_try_advisory_lock(12345);
   ```

1.3.2. **获取锁并执行任务（推荐事务级）**：

   ```sql
   BEGIN;
   -- 获取事务级锁，如果已被占用则等待
   SELECT pg_advisory_xact_lock(12345);
   -- 执行你的业务逻辑...
   COMMIT; -- 提交后锁自动释放
   ```

1.3.3. 为什么选择 PG 替代 Redis？

- **简化架构**：如果你的应用已经在使用 PG，无需为了简单的分布式锁引入 Redis 这种额外的基础设施，降低运维压力。
- **安全性更高**：咨询锁不占用表行，不会导致表膨胀或索引问题，且由数据库内核保证原子性。
- **内置监控**：可以通过 `pg_locks` 视图实时监控哪些进程持有锁，排查问题非常方便。

------

## 1.4. **总结**

对于绝大多数业务场景，使用 PG 的 `pg_advisory_xact_lock` 是替代 Redis 分布式锁的最优选，它既保证了安全性，又简化了代码中的“续期”与“防死锁”逻辑。

咨询锁的问题：
1. 无法快速感知锁状态，不够直观，不能像 redis 锁一样查询 key 快速确认某项业务是否存在锁，而且吞吐量不如 redis 锁，并发数量上受限于数据库连接池。
2. 使用同一个业务锁的服务只能连接同一个库，否则会锁不住，因为咨询锁不会同步到其它节点，而且所连的库挂掉锁也会丢失，如果在事务中有耗时网络任务就无法立刻得到反馈。

扩展内容：pg_try_advisory_xact_lock

# 3. 时序库（TimescaleDB）

## 3.1. PG 扩展（Extension）的特性
- **非数据库独立**：扩展是深度绑定 PG 架构的（利用 C Hook、操作符等接口），无法直接在 MySQL/Oracle 运行。
- **数据库级安装**：在同一实例下，扩展需按数据库（Database）单独安装，哪个库需要就单独启用。
- **生态优势**：通过 PostGIS（地理）、PGVector（向量）、TimescaleDB（时序）等，PG 可实现“一专多能”。

```sql
# 查询 timescaledb 是否可用
SELECT name, default_version, installed_version 
FROM pg_available_extensions 
WHERE name = 'timescaledb';

# 查询 timescaledb 是否已启用
SELECT * FROM pg_extension WHERE extname = 'timescaledb';

# 查看共享库中是否已包含 timescaledb，没有的话要么启用不了，要么启用时会失败，共享库启用需要重启数据库
SHOW shared_preload_libraries;

# 启用 timescaledb 扩展
CREATE EXTENSION IF NOT EXISTS "timescaledb" CASCADE;
```

## 3.2. 时序库管理亿级数据的核心机制

面对亿级数据，传统 PG 单表会因索引过大导致 I/O 崩溃。TimescaleDB 通过以下手段解决：

1. **超表 (Hypertables)**：自动将大表按时间切分为物理分片（Chunks），可以让写入始终集中在新的 Chunk，保证索引始终能放入内存。
2. **列式压缩 (Compression)**：将行存转为列存，存储空间减少 90% 以上，显著提升历史数据的范围聚合速度。
3. **持续聚合 (Continuous Aggregates)**：可以自动预算汇总数据（如日均、月均），查询报表时无需扫描原始亿级行，亚秒级响应。

**相比于传统分区表，时序表的优点在于无须手动编写复杂的分区表创建与管理逻辑，只要数据是根据时间字段进行管理，那就能自动分片；但缺点也很明显，如果没有时间列就无法使用。**

<span style="color: red;">**注意：持续聚合特性需要2.x版本开始才能有较好的支持，在那之前的使用方式是有所不同的。**  </span>

超表的定义方式：
```sql
-- 首先，超表一定需要时间字段，配合 chunk_time_interval 参数对数据按时间周期进行分块存储，数据多可以一天一片，数据少可以一月一片
-- 其次，partitioning_column 和 number_partitions 可以指定额外的分块逻辑（空间分块），通常应指定为CPU核心倍数，和时间片是相乘关系
-- 最后，创建了超表会自动创建时间索引，如果有分区列还会创建分区列与时间列的复合索引。分区列不建议超过一个，否则分块数量爆炸也会影响查询
SELECT create_hypertable(
   '表名',
   '时间字段',
   partitioning_column => '额外的分片字段',
   number_partitions => 8,
   chunk_time_interval => INTERVAL '1 day'
);
```
> 假如原表有个 type 字段具有 12 种类型，被指定为分区列并设置了8个分区，那么这些类型就会被哈希算法分配到这8个分区里，有的保存 1 种类型，有的保存 2 种类型。因为物理上这些数据被分配到了不同块上，只要查询时能带上时间和分区条件，查询范围将会被限制到最小。

持续聚合的定义：

```sql
-- timescaledb 1.x 用普通视图，2.x 用物化视图，不过各版本变化较多，实际创建时可以都尝试下
--   1.x 无法直接修改定义中的 SQL，但可以修改 WITH 定义中的参数，
--   如果要改 SQL 需要 DROP VIEW 重新 CREATE VIEW，这会导致统计结果丢失
-- WITH (timescaledb.continuous)：这是核心开关，告诉 TimescaleDB：“这不是普通的视图，请为它创建一个后台物理表来存结果”。
-- timescaledb.refresh_interval：刷新周期。指后台任务每隔多久运行一次（例如每小时算一次）。
-- timescaledb.refresh_lag：刷新延迟（水位线）。指刷新时忽略最近多久的数据。
--   因为时序数据常有延迟（网络抖动等），如果立即聚合，可能会漏掉还没跑到的数据。
--   设为 1h 表示现在只计算 1 小时前的数据，确保数据到齐。
-- time_bucket：必须存在。它定义了聚合的“颗粒度”（桶大小）。
CREATE VIEW device_summary_hourly 
WITH (
    timescaledb.continuous, 
    timescaledb.refresh_interval = '1h', 
    timescaledb.refresh_lag = '1h'
) 
AS
SELECT 
    time_bucket('1 hour', "time") AS bucket,
    device_id,
    avg(temperature) AS avg_temp
FROM sensor_data
GROUP BY bucket, device_id;
-- 如果想要实现聚合数据实时更新，那么可以修改视图配置，
-- 这样 timescaledb 会联合视图数据和主表数据统计合并后返回，
ALTER VIEW device_summary_hourly SET (timescaledb.materialized_only = false);
-- 如果太久的数据进入可能不会重算，手动触发内部存储过程可以指定补录时间段
-- 例如：手动刷新指定的时间区间（假设是 3 天前到 1 天前的数据）
SELECT _timescaledb_internal.refresh_continuous_aggregate(
    'device_summary_hourly'::regclass, -- 你的聚合视图名称
    '2023-10-01 00:00:00'::timestamptz, -- 窗口起点（必须早于补报数据的时间）
    '2023-10-03 00:00:00'::timestamptz  -- 窗口终点
);
-- 修改视图的刷新周期
ALTER VIEW device_summary_hourly SET (timescaledb.refresh_interval = '30m');
-- 手动触发持续聚合视图刷新
REFRESH MATERIALIZED VIEW device_summary_hourly;
-- 查看持续聚合执行信息
SELECT * FROM timescaledb_information.continuous_aggregates;
```
> 数据聚合这件事非常看业务，由于 1.x 时序表聚合修改难度大，所以最好一开始就定好业务聚合方式。如果觉得低版本时序表灵活性不足，那么可以考虑用自定义的聚合表来替代时序表的持续聚合。
>
> 对于一些非整点时区进行统计时，时序库提供的`time_bucket`函数能够简化非常多的时间处理逻辑。例如，尼泊尔（`+05:45`）这种特殊时区，进行统计时可以有两种方式格式化时间（以尼泊尔时间`2024-01-01 00:00:00`作为原始时间参考）：
>
> 1. 使用` time_bucket('1 hour', time AT TIME ZONE 'Asia/Kathmandu') AS local_hour`可以格式化为当地时间，输出`2024-01-01 00:00:00`。
> 2. 使用`time_bucket('1 hour', time, INTERVAL '45 minutes') AS shifted_hour`可以格式化为带时区的时间，输出`2023-12-31 18:15:00+00:00`。
> 
> 从灵活性上来说第二种方式更好，但第一种方式可以直接将数据输出到展示页面，对于时区固定的情况比较方便。

## 3.3. 历史数据补录 (Backfilling) 
前面提到数据一旦压缩就不太好修改，这个修改包含两部分内容，一部分是新的历史数据，另一部分是旧数据的修改，这里要讨论的内容同时涉及这两方面。通常来说补录的处理策略如下：

### 3.3.1.  零星数量补录
- **直接写入**：TimescaleDB 会自动找到对应历史时间的 Chunk 进行解压后再处理。
- **兼容性**：新版本（2.10+）才支持直接向已压缩的 Chunk 写入数据。

<span style="color: red;">**需要注意，如果定义的时间块比较大，那么为了降低解压开销，可以考虑将时间周期减小，例如从一月一块减小到一天一块。**</span>

### 3.3.2.  持续小规模补录

- **拉长压缩周期**：确保自动压缩周期内不会再有数据补录或不再接收超期数据时开始压缩
- **冷热表分离**：将超过自动压缩周期的数据放入冷表，等凌晨再统一发起解压、补录与压缩

自动压缩周期最好不要拉太长，否则过多数据无法压缩，会丧失列式压缩带来的性能优势。使用冷热表时，需要注意查询时应通过 UNION ALL 将两表的数据结合起来。

对于旧数据修改的情况，如果有条件的话可以考虑通过版本管理模式实现补录，增加**版本号**或**修改时间**字段，将修改后的行作为新数据插入，查询时只返回最新那条。

或者增加附表用于存放修改的列信息，查询时先将需要的数据查出来，再根据主键从附表中获取修改的列信息进行合并。


### 3.3.3. 大规模补录流程（性能最优）

1. **解压**：针对目标历史区间执行 `decompress_chunk()`。
2. **导入/修改**：使用 `COPY` 命令或并行导入工具或执行批量更新操作。
3. **重压缩**：导入或修改完成后重新执行 `compress_chunk()`。
4. **同步视图**：调用 `refresh_continuous_aggregate()` 刷新受影响的聚合报表。

## 3.4. 大文本列（TEXT/JSONB）专项优化
当表中含有 `TEXT` 等大数据列时，需防止其拖慢主表扫描速度。


| 优化手段     | 操作方法                                                | 核心目的                                                     |
| :----------- | :------------------------------------------------------ | :----------------------------------------------------------- |
| **存储模式** | `ALTER TABLE ... ALTER COLUMN ... SET STORAGE EXTERNAL` | 将大文本强制行外存储，保持主表“苗条”。                       |
| **分区粒度** | 缩短 `chunk_time_interval` (如设为 1 天)                | 防止包含大文本的单个物理 Chunk 过大（控制在 10-20GB）。      |
| **分表策略** | 垂直拆分：指标表（Thin）+ 详情表（Thick）               | 90% 的查询只读指标表，避免加载大文本，具体查询时才从详情表获取大数据列内容。 |
| **检索优化** | 使用 `GIN` 索引 + `pg_trgm`                             | 解决亿级文本下的模糊匹配（LIKE）性能问题。                   |

对整个大数据列应用`GIN`索引可能会导致空间膨胀，极端情况下可能比原表占用还大，所以最好是根据需要提取大数据列中的数据字段建立专用索引，如果对于性能要求较高可以考虑使用生成列（仅限PG12开始的版本可用，之前的版本需要使用触发器模拟这个功能）提取为物理存在的列数据。
> 在 PostgreSQL 中，生成列（Generated Column） 是一种特殊的列，它的值始终由表中其他列计算而来。你可以把它理解为数据库层面的“自动公式”。
>
> 目前 PostgreSQL（从 v12 开始）仅支持 STORED（存储式） 生成列，即计算结果会物理存储在磁盘上。
>
> 生成列通过 `GENERATED ALWAYS AS (表达式) STORED` 来定义。它具有以下特性：
>
> - **自动计算**：插入或更新原始列时，该列自动重新计算。
> - **物理存储**：占用磁盘空间，但查询速度和普通列一样快。
> - **不可手动修改**：你不能直接对生成列进行 `INSERT` 或 `UPDATE`。

---

## 3.5. 简要示例

### 3.5.1. 建表与初始化
```sql
CREATE TABLE sensor_data (
    time        TIMESTAMPTZ NOT NULL,
    device_id   INT NOT NULL,
    cpu_usage   DOUBLE PRECISION,
    payload     TEXT -- 大文本列
);

-- 1. 创建超表
SELECT create_hypertable('sensor_data', 'time', chunk_time_interval => INTERVAL '1 days');

-- 2. 优化大文本列存储，大小远超2KB时使用外部存储方式，如果能分离为附表更佳
ALTER TABLE sensor_data ALTER COLUMN payload SET STORAGE EXTERNAL;
```

### 3.5.2. 配置自动化策略


```sql
-- 开启压缩，按设备 ID 分组以提升压缩率
ALTER TABLE sensor_data SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'device_id'
);

-- 自动压缩 14 天前的数据
SELECT add_compression_policy('sensor_data', INTERVAL '14 days');

-- 自动删除 365 天前的数据（保留策略）
SELECT add_retention_policy('sensor_data', INTERVAL '365 days');
```

### 3.5.3. 查询加速：创建持续聚合

如果你需要查询“每日平均温度”等汇总信息，不要直接扫亿级大表，使用持续聚合视图 [3, 4]。

```sql
-- 创建持续聚合视图（相当于物化视图，但会自动增量更新）
CREATE MATERIALIZED VIEW sensor_day_stats
WITH (timescaledb.continuous) AS
SELECT time_bucket('1 day', time) AS day,
       device_id,
       AVG(cpu_usage) as avg_cpu,
       MAX(temperature) as max_temp
FROM sensor_data
GROUP BY day, device_id;

-- 设置自动刷新策略：每天刷新一次过去 3 天的数据
SELECT add_continuous_aggregate_policy('sensor_day_stats',
    start_offset => INTERVAL '3 days',
    end_offset => INTERVAL '1 hour',
    schedule_interval => INTERVAL '1 day');
```

### 3.5.4. 核心场景：如何补录历史数据

补录历史数据时，由于数据可能落入**已压缩**的块或**已完成计算**的聚合视图中，需要特殊处理。

#### 步骤 1：向已压缩的时间段插入数据

在 TimescaleDB 2.11+ 版本中，可以直接执行 `INSERT`。但对于**亿级规模**的补录，建议先解压以获得最高写入速度 [5, 6]。

```sql
-- 如果补录范围很大（例如补录 2024 年 1 月的数据）
-- 1. 手动解压特定时间段的 Chunk
SELECT decompress_chunk(c, true) 
FROM show_chunks('sensor_data', '2024-01-01', '2024-01-31') c;

-- 2. 批量导入数据（建议使用 COPY 命令）
-- COPY sensor_data FROM 'data_2024_01.csv' WITH (FORMAT CSV);

-- 3. 补录完成后重新压缩
SELECT compress_chunk(c, true) 
FROM show_chunks('sensor_data', '2024-01-01', '2024-01-31') c;
```

#### 步骤 2：更新补录后的聚合报表

插入历史数据后，持续聚合视图不会自动回溯太远的时间。你需要手动触发一次刷新，确保报表准确 [7, 8]。

```sql
-- 手动刷新指定历史时间段的聚合数据
CALL refresh_continuous_aggregate(
    'sensor_day_stats', 
    '2024-01-01', 
    '2024-02-01'
);
```

### 3.5.5. 总结建议

| 操作           | 建议                                                         |
| :------------- | :----------------------------------------------------------- |
| **日常写入**   | 直接 `INSERT` 即可，TimescaleDB 会自动路由到最新的 Chunk。   |
| **小规模补录** | 直接插入。2.11 后的版本支持在压缩 Chunk 上直接增删改 [5]。   |
| **大规模补录** | **先解压 -> 批量导入 -> 重压缩 -> 刷新持续聚合**，这是性能最优解。 |
| **查询历史**   | 尽量查询**持续聚合视图**而非原始超表，速度可提升百倍。       |

## 3.6. 总结的总结

管理亿级 PG 数据的关键在于：**分而治之（Hypertable）**、**能省则省（Compression）**、**提前计算（Continuous Aggregates）**。对于大文本，应通过**存储模式调整**确保其不成为查询性能的“拖油瓶”。

# 4. TEXT、JSON与JSONB

简单来说，这三者在 PostgreSQL 中代表了从“纯文本”到“结构化二进制”的进化。

以下是它们的直观对比：

## 4.1. 核心区别对照表

| 特性           | **TEXT**             | **JSON**                  | **JSONB** (推荐)              |
| :------------- | :------------------- | :------------------------ | :---------------------------- |
| **存储格式**   | 纯字符串             | 完整文本拷贝              | **解析后的二进制格式**        |
| **写入速度**   | 极快 (无校验)        | 快 (仅验证语法)           | 略慢 (需转换成二进制)         |
| **查询速度**   | 慢 (需正则/全表扫描) | 慢 (每次查询都要重新解析) | **极快** (直接定位 key/value) |
| **索引支持**   | 全文检索/前缀索引    | 极受限                    | **强大 (支持 GIN 索引)**      |
| **空格与顺序** | 保留原始格式         | 保留原始格式              | 去除多余空格，不保留 key 顺序 |
| **重复 Key**   | 保留所有             | 保留所有                  | **自动去重** (保留最后一个)   |

------

## 4.2. 详细解读

**TEXT：原始载体**

- **本质**：就是一段没格式的长字符串。
- **缺点**：数据库完全不理解内容。如果你想查询 `where info ->> 'name' = '张三'`，数据库必须把整段文字读出来再用代码逻辑判断，性能极差。

**JSON：纯文本存储 (Legacy)**

- **本质**：带格式校验的 TEXT。
- **特点**：它会检查你存进去的是不是合法的 JSON 格式。
- **场景**：如果你只需要原封不动地存储并原样取出（例如：前端传什么我就存什么，连空格都要一模一样），可以使用它。但这种场景极少。

**JSONB：结构化二进制 (The Winner)**

- **本质**：**B** 代表 **Binary**。它在入库时就把 JSON 拆解成了树状的二进制结构。
- **优势**：
  - **查询飞快**：寻找某个 key 时不需要从头读字符串，而是直接通过偏移量定位。
  - **GIN 索引**：你可以给整个 JSONB 字段加一个 GIN 索引。哪怕里面有几百个 key，查询其中任何一个 key 的速度都是毫秒级的。
  - **处理能力**：支持多种内置操作符（如 `@>` 包含关系、`?` 键是否存在等）。

------

## 4.3. 选型建议

1. **绝大多数场景直接选 `JSONB`**：它是 PG 的明星特性，支持高效索引和亚秒级查询。
2. **只有在以下极端情况选 `JSON`**：
   - 你对写入性能有极致要求，且几乎从不在数据库层面查询内部字段。
   - 必须严格保留 JSON 原始的空格、缩进和 key 的重复性。
3. **不要选 `TEXT` 存储 JSON**：除非你的 PG 版本太低（低于 9.2），否则没有任何理由用 TEXT 存结构化数据，因为它缺乏语法校验且查询效率最低。

**总结一句话：** 只要你打算在 SQL 里用到 `->` 或 `->>` 这种操作符来查数据，请闭眼选 **`JSONB`**。

## 4.4. 支持情况

只要你的 PostgreSQL 版本在 **9.4**（2014年发布）或更高，你无需安装任何扩展（Extension）或进行额外配置，就可以直接在建表时使用 `JSONB` 类型。

关键特性

- **版本支持**：PostgreSQL 9.2 引入了基础的 `JSON` 类型（仅文本校验），而更强大的 `JSONB`（二进制存储，支持索引）在 **9.4 版本** 正式加入。

- **无需扩展**：与需要手动执行 `CREATE EXTENSION` 的插件（如 `postgis` 或 `timescaledb`）不同，`JSONB` 是核心代码库的一部分。

- **使用方式**：直接在 `CREATE TABLE` 语句中定义即可：

    ```sql
    CREATE TABLE example (
      id serial PRIMARY KEY,
      data JSONB  -- 直接使用，无需额外操作
    );
    ```

为什么说它比 JSON “更好”？

虽然 `JSON` 和 `JSONB` 都是内置的，但业界普遍建议将 `JSONB` 作为默认选择：

- **性能**：`JSONB` 以二进制格式存储，虽然写入时有微小的转换开销，但查询时不需要重新解析，速度远快于 `JSON`。

- **索引支持**：`JSONB` 支持 GIN（通用反向索引），可以对 JSON 内部的键值对进行高效检索。

- **灵活性**：它会自动去除多余空格并压缩存储，且不保留键的重复项。

## 4.5. 大表优化

在处理上亿级（100M+）数据量时，`TEXT`、`JSON` 和 `JSONB` 的性能表现差异巨大。由于数据量极大，优化核心在于减少 I/O 扫描和**索引效率**。

以下是针对这三种类型在 TimescaleDB/PostgreSQL 亿级场景下的专项优化方案：

### 4.5.1. TEXT

作为“原始载体”的优化。在亿级数据中，`TEXT` 通常用于存储日志或描述。

- **优化手段**：`SET STORAGE EXTERNAL`。
- **原理**：强制将长文本移出主表（TOAST 存储）。这样在进行 `COUNT(*)` 或按时间范围聚合数值时，数据库不需要跳过巨大的文本块，扫描速度提升 5-10 倍。
- **搜索**：禁止使用 `LIKE '%word%'`。必须使用 `pg_trgm` 扩展建立 **GIN 索引**。

### 4.5.2. JSON

最不推荐的亿级选项

- **问题**：`JSON` 每次查询都要重新解析整个字符串。在亿级数据下，任何涉及内部字段的查询都会导致 CPU 爆满。
- **优化建议**：**将其转换为 `JSONB` 或 `TEXT`**。
  - 如果你不需要查询内部字段，用 `TEXT` 并配合 `STORAGE EXTERNAL`。
  - 如果你需要查询内部字段，立即迁移到 `JSONB`。

### 4.5.3. JSONB

亿级结构化数据的“王牌”。`JSONB` 是处理非结构化亿级数据的首选，但如果不优化，索引会比数据还大。

A. 索引瘦身：表达式索引 (Expression Index)

不要对整个 `JSONB` 字段建 GIN 索引（索引会非常臃肿）。

- **优化**：只针对高频查询的键建索引。

  ```sql
  -- 仅对 JSONB 内部的 device_id 建索引，而不是整个 JSONB
  CREATE INDEX idx_json_device ON sensor_data ((payload->>'device_id'));
  ```

B. 局部 GIN 索引 (Partial GIN Index)

- **优化**：如果你只关心特定类型的 JSON 数据，可以建立带 `WHERE` 条件的索引。

  ```sql
  CREATE INDEX idx_error_logs ON sensor_data USING GIN (payload) 
  WHERE (payload->>'status' = 'error');
  ```

C. 列式压缩与 JSONB

TimescaleDB 的压缩对 `JSONB` 非常友好。

- **优化**：开启 TimescaleDB 压缩。它能识别 `JSONB` 中的重复键名，并通过列式存储大幅降低冗余，压缩率通常能达到 80% 以上。

------

## 4.6. 综合对比与选型指南 (亿级规模)

| 类型      | 存储占用      | 查询性能 | 推荐优化方案                                                 |
| :-------- | :------------ | :------- | :----------------------------------------------------------- |
| **TEXT**  | 中            | 极低     | `STORAGE EXTERNAL` + `GIN Trigram` 索引                      |
| **JSON**  | 高            | 极低     | **不建议在亿级规模使用**，应转为 JSONB                       |
| **JSONB** | 极高 (索引大) | **极高** | **1. 开启 TimescaleDB 压缩** **2. 使用表达式索引而非全量 GIN** **3. 缩短超表分区区间 (Chunk Interval)** |

专家建议：垂直拆分 (The "Gold" Standard)

对于亿级数据，最极致的优化是**“动静分离”**：

1. **超表 (主表)**：只存放 `time`, `id`, `value` 等固定格式的数值。
2. **详情表 (JSONB 表)**：存放不规则的 `JSONB` 数据。
3. **查询策略**：99% 的时序分析只查主表。只有当用户点击“查看详情”时，再通过 `time` 和 `id` 去 JSONB 表中提取单行数据。

**总结：** 在亿级场景下，**JSONB + TimescaleDB 压缩 + 局部索引** 是兼顾灵活性与性能的最佳组合。

# 5. 数据迁移

## 5.1. 导出

### 5.1.1. pg_dump

这是 PG 的核心备份工具，支持**在线备份**（本质其实是执行`SELECT`然后将数据拼接成所需的文件格式，所以备份时不影响数据库读写），备份文件的可移植性高。

`pg_dump` 导出的文件主要有三种格式，通过 `-F` 参数指定：

- **SQL 脚本 (`-Fp`)**：默认格式。生成纯文本 `.sql` 文件。优点是可读性强，可以用记事本打开修改；缺点是恢复速度慢，不支持并行。
- **自定义压缩 (`-Fc`)**：**最推荐的格式**。生成二进制压缩文件（通常后缀为 `.dump` 或 `.bak`）。优点是文件小、恢复速度快，且配合 `pg_restore` 可以灵活选择只恢复某张表。
- **目录格式 (`-Fd`)**：生成一个文件夹。优点是支持**并行导出**（多线程），适合 TB 级超大型数据库。

------

常用操作命令示例：

A. 导出整个数据库（最常用），导出结果为二进制压缩格式，体积小便于传输，恢复速度也快。

```bash
pg_dump -U 用户名 -d 数据库名 -Fc -f backup_file.dump
```

B. 导出为纯 SQL 脚本，可以直接看到建表语句和 `INSERT` 语句。

```bash
pg_dump -U 用户名 -d 数据库名 -f backup.sql
```

C. 只导出特定的表，适合只想迁移部分数据的场景。

```bash
pg_dump -U 用户名 -d 数据库名 -t 表名1 -t 表名2 -Fc -f tables_backup.dump
```

D. 只导出表结构（使用`-s`参数，不带数据），适合初始化新环境。

```bash
pg_dump -U 用户名 -d 数据库名 -s -f schema_only.sql
```

E. 只导出数据（使用`-a`参数，不带结构）

```bash
pg_dump -U 用户名 -d 数据库名 -a -f data_only.sql
```

F. 使用连接字符串进行导出，相比于其他方式来说最大的好处是可以做到免输密码。

```bash
pg_dump "postgresql://用户名:密码@数据库地址:端口/数据库名" -Fc -f backup_file.dump
```

> 连接字符串除了 URI 结构外还有一种键值对结构：
>
> `"host=localhost port=5432 user=myuser password=mypassword dbname=db1"`
>
> 如果前缀是`postgres://`或`postgresql://`开头则作为 URI 连接串解析，否则按照键值对解析，都不行就当作单个库名使用。
>
> 如果库名有等号（`=`），那么要使用 URL 连接串，然后将等号使用 URL 参数转义（`%3D`）。

G. 一行指令导出纯数据的 SQL 文件并压缩为 gzip 类型文件。

```bash
pg_dump "postgresql://用户名:密码@数据库地址:端口/数据库名" -a | gzip > data_only.sql.gz
```

------

一些关键参数说明：

- **`-h（--host）` / `-p（--port）`**：指定主机地址和端口（默认 5432）。
- **`-U（--username）` / `-W（--password）`**：指定用户名和强制提示输入密码。
- **`-d（--dbname）`**：指定数据库名称，可以省略只写一个字符串，如果指定了完整的连接串可以省略`-h`、`-p`等参数。
- **`-t（--table） <表名>`**：指定导出的表，可以指定多个，配合`-a`或`-s`可以分别只导数据或只导结构。
- **`-j（--jobs） <线程数>`**：**并行导出**。仅限目录格式 (`-Fd`)，能极大提升大数据量的导出效率。
- **`-Z（--compress） <压缩级别>`**：指定压缩等级（0-9，默认 6），值越大压缩率越高，也越耗费 CPU。在 PG 16+ 上配合`-Fp`使用时 SQL 文件会被压缩为 gzip（.gz）文件。
- **`-F（--format） <格式参数>`**：指定导出格式（参数后紧跟），主要有三种格式，分别为`p`（`plain`，SQL文件）、`c`（`custom`，二进制单文件）和`d`（`directory`，目录多文件）。
- **`-a（--data-only）`**：只导出数据，不带结构。
- **`-s（--schema-only）`**：只导出结构，不带数据。
- **`-n（--schema） <模式名>`**：只导出某个 Schema（模式）下的内容。
- **`-T（--exclude-table） <表名>`**：排除某些大表（比如日志表）不备份。
- **`-N（--exclude-schema） <模式名>`**：排除某些模式不备份。
- **`-c（--clean）`**：增加`DROP`语句用于导入前进行清理。
- **`-C（--create）`**：包含建库的语句。
- **`-x（--no-privileges）`**：无视复杂的权限分配（防止因为目标环境角色不存在而报错）。
- **`--no-tablespaces`**：忽略所有表空间归属的定义，重新导入时会作为默认表空间的数据导入。
- **`--no-owner`**：导出时不带所有者信息。这在**跨用户还原**时非常有用，避免还原时报错找不到原用户。
- **`--inserts`**：导出时使用`INSERT`语句而不是`COPY`语句（通常用于跨不同类型数据库，但数据较慢）。
- **`--on-conflict-do-nothing`**：如果你想在导入时自动跳过已存在的重复数据（需配合 `--inserts`）。
- **`--disable-triggers`**：仅限导出 plain 类型文件时使用，会添加禁用触发器语句，可以提高导入效率。

> pg_dump 导出时只记录表和索引所属的表空间，并不会产生表空间的定义配置，只要新库有这个表空间就行。
>
> 如果不想再让表和索引受表空间约束，可以使用`--no-tablespaces`忽略所有表空间归属定义。
>
> 如果导出的类型为 plain ，那么也可以通过手动修改 SQL 语句来修改表空间归属。

------

**密码处理：**

`pg_dump` 执行时通常会提示输入密码。为了自动化脚本运行，可以使用以下方式避开手动输入：

1. **环境变量**：`export PGPASSWORD='你的密码'` (不安全，历史记录可见)。

2. **密码文件**：在用户主目录创建 `.pgpass` 文件（**最推荐**），且权限设置为`0600`才会生效（使用`chmod 0600 ~/.pgpass`赋权）。

   > 密码文件内每行存储一条密码记录，需要指定数据库地址、端口、库名、用户名和密码，结构为`hostname:port:database:username:password`。
   >
   > 运行时根据 `-h`, `-p`, `-d`, `-U` 参数从上到下搜寻符合的记录，找到底一条满足的记录后就停止搜索。

3. **连接串**：在`--dbname`参数中使用完整连接串并指定密码，例如`pg_dump "postgresql://用户名:密码@数据库地址:端口/数据库名" -a -Fc -Z 9 -f data_only.dump`。

### 5.1.2. pg_dumpall

相比于`pg_dump`只是导出一个库，`pg_dumpall`用于导出整个 PG 实例的所有库。在导出格式上，`pg_dumpall`只能导出为 SQL 文件，因此无法通过指定`-Fc`参数导出二进制格式。

使用举例：导出全局对象到 SQL 文件中。

```sql
pg_dumpall -g > roles_backup.sql
```

------

一些关键参数说明：

- **`-h（--host）` / `-p（--port）`**：指定主机地址和端口（默认 5432）。
- **`-U（--username）` / `-W（--password）`**：指定用户名和强制提示输入密码。
- **`-d（--dbname）`**：指定要连接的库，也支持连接字符串。**需要注意，这个库只是作为连接使用，并不是只导出这一个库。**
- **`-l（--database）`**：与`-d`作用一致，属于早期参数，不如`-d`可使用连接字符串那么灵活。
- **`-a（--data-only）`**：只导出数据，不带结构。
- **`-s（--schema-only）`**：只导出结构，不带数据。
- **`-T（--exclude-table） <表名>`**：排除某些大表（比如日志表）不备份。
- **`-N（--exclude-schema） <模式名>`**：排除某些模式不备份。
- **`-c（--clean）`**：增加`DROP`语句用于导入前进行清理。
- **`--no-owner`**：导出时不带所有者信息。这在**跨用户还原**时非常有用，避免还原时报错找不到原用户。
- **`-g（--globals-only）`**：最常用。 只导出全局对象（用户、角色、表空间），不导出任何具体的数据库数据
- **`-r（--roles-only）`**：只导出角色（用户和组），不包含表空间和数据。
- **`-t（--tablespaces-only）`**：只导出表空间的定义语句。
- **`-x（--no-privileges）`**：无视复杂的权限分配（防止因为目标环境角色不存在而报错）。
- **`--no-tablespaces`**：忽略所有表空间定义，重新导入时会作为默认表空间的数据导入。

> **表空间**是个特殊的资源配置项，当你导出时携带表空间，对于表空间指定的物理路径需要人工保障。
>
> 假如导出的语句为：`CREATE TABLESPACE my_space LOCATION '/data/pg_tbs/my_space';`。
>
> 那么目标数据库所在服务器上的路径需要手动创建，并且可用空间也需要做好保障。
>
> 如果没有办法保持原路径，那么也可以通过手动修改 SQL 语句来指定新的路径。

### 5.1.3. copy

1. **`COPY`是 PG 中性能最高的导出方式，它使用的是流式导出，直接将原始数据块格式化后输出，针对海量数据导出有奇效，但它只能导数据，这是它最大的局限性。**

它的导出语法结构为：

```sql
COPY { 表名 [ ( 列名 [, ...] ) ] | ( 查询语句 ) }
    TO { '文件绝对路径' | PROGRAM '外部程序命令' | STDOUT }
    [ WITH ( 选项 [, ...] ) ]
```

`COPY`有两种使用方式：

- 服务端执行并导出到服务端：`COPY my_table TO '/tmp/data.csv' WITH (FORMAT CSV, HEADER);`。这种方式要通过**数据库超管**执行 SQL 语句来完成，文件只会生成到服务端。
- 客户端执行并导出到客户端：`\copy my_table TO 'C:/Users/Desktop/data.csv' WITH (FORMAT CSV, HEADER)`。需要用户在使用`psql`登录后再执行，文件可以保存到用户本地。

除了整表导出外，`COPY`也可以使用子查询来导出部分数据：

```sql
COPY (SELECT id, name FROM users WHERE age > 18) TO '/tmp/adults.csv' WITH (FORMAT CSV);
```

虽然子查询的方式灵活，但是这种方式会导致流式导出的效率有明显下降，查询逻辑越复杂速度越慢受影响越大。如果业务上能接受导全表就不要用子查询。

如果非要用子查询，那么尽量使用全列查询方式，如果有分块的需求，也应该在子查询中进行分割：

```sql
COPY (SELECT * FROM table LIMIT 1000000 OFFSET 0) TO 'part1.csv' WITH (FORMAT CSV);
COPY (SELECT * FROM table LIMIT 1000000 OFFSET 1000000) TO 'part2.csv' WITH (FORMAT CSV);
```

------

2. **`TO`子句的用法**

`TO`子句允许你将数据传递给三种不同的目标：

- **文件**：服务端方式执行则文件在服务端生成，客户端方式执行则文件在服务端生成。服务端方式需要数据库**超级管理员**的权限才能执行这种目标的调用，文件路径权限受系统用户管控。

- **程序（PRORAM）**：可以调用数据库进程有权限的程序执行数据传递，需要数据库**超级管理员**的权限才能执行这种目标的调用。

  > 语法中的 `PROGRAM` 作为 PG 的**高级黑科技**，允许你直接把数据传给 Linux 命令处理（数据库进程的执行用户需要有权限调用传递给`PROGRAM`的程序）。
  > 
  > - **边导出边压缩**：
  > 
  >   ```sql
  >   COPY my_table TO PROGRAM 'gzip > /tmp/data.csv.gz' WITH (FORMAT CSV);
  >   ```
  > 
  > - **直接传给外部脚本**：数据流会通过管道直接喂给你的 Python 或 Shell 脚本，无需中间落地。
  > 
  >   ```sql
  >   COPY my_table TO PROGRAM 'python3 /path/to/process_data.py' WITH (FORMAT CSV);
  >   ```
  > 
  > - **边导边传**：直接通过 curl 把数据传给远程 API。
  >   
  >   ```sql
  >   COPY my_table TO PROGRAM 'curl -X POST --data-binary @- http://my-api/upload' WITH (FORMAT CSV);
  >   ```
  > 
  > - **动态脱敏**：在数据写出前用正则把手机号遮掩掉。
  >   
  >   ```sql
  >   COPY my_table TO PROGRAM 'sed "s/13[0-9]\{9\}/XXXXX/g"' WITH (FORMAT CSV);
  >   ```

- **标准流（STDOUT）**：没有用户权限的要求，数据会通过网络通道直接输出，在`JDBC`驱动中有专门的调用方式。

  > ```java
  > // 1. 将常规连接转换为 PG 的基础连接
  > CopyManager copyManager = new CopyManager((BaseConnection) conn);
  > 
  > // 2. 执行 COPY 命令并返回一个 InputStream
  > // 数据库此时开始通过网络流式推送数据
  > InputStream out = copyManager.copyOut("COPY my_table TO STDOUT WITH (FORMAT CSV)");
  > 
  > // 3. 像处理文件一样展示数据
  > try (BufferedReader reader = new BufferedReader(new InputStreamReader(out))) {
  >     String line;
  >     while ((line = reader.readLine()) != null) {
  >         // 在这里展示数据，比如打印到控制台
  >         System.out.println("收到行数据: " + line);
  >     }
  > }
  > ```

-------

3. **深度拆解：`WITH` 子句中的关键选项（该子句非必须，选项的内容决定了数据的“长相”和处理逻辑）**

- **`FORMAT`**: 指定数据格式。
  - `text` (默认)：简单的纯文本。
  - `csv`：标准逗号分隔值格式。
  - `binary`：二进制格式。**速度最快**，但不可读，且跨版本兼容性略差。
- **`DELIMITER`**: 指定分隔符（仅文本/CSV）。比如 `DELIMITER '|'`。
- **`HEADER`**: 仅用于 CSV。导出时包含表头，导入时跳过第一行。
- **`QUOTE` / `ESCAPE`**: 仅用于 CSV。定义引用符（默认`"`）和转义符。
- **`NULL`**: 指定空值的表示字符串。默认文本格式下是 `\N`，CSV 格式下是空字符串。
- **`ENCODING`**: 指定文件编码（如 `UTF8`, `GBK`）。

------

4. **流式导出的原理：为什么这么快？**

`COPY` 是 PostgreSQL 的“性能怪兽”，它的核心在于绕过了复杂的 SQL 解析和事务开销。

普通的 `SELECT` 查询会经历：**解析 -> 优化 -> 重写 -> 执行 -> 格式化输出 -> 逐行返回客户端**。而 `COPY` 的“流式”原理如下：

A. 绕过 SQL 层级

`COPY` 本质上是一个特殊的执行节点。它不生成复杂的执行计划，而是直接通过**表句柄**访问底层的 **堆存储 (Heap Storage)**。

B. 原始数据块处理 (Bulk Transfer)

- **读取层面**：它以 **8KB 的数据块 (Page)** 为单位进行预读。
- **转换层面**：它不需要将数据转换成复杂的“结果集”对象，而是直接将字段的二进制值推入发送缓冲区。
- **网络/磁盘协议**：它使用专用的 **CopyData 协议**。数据在网络上是连贯的字节流，没有查询结果集中的那些元数据包（Metadata Packet）干扰，开销降到了最低。

C. “零拷贝”思想 (针对二进制模式)

在 `FORMAT BINARY` 模式下，PG 甚至不进行字段的文本转换，直接把内存里的底层存储表示发送出去。这种方式几乎只受限于磁盘 I/O 和网络带宽。

-------

**总结：**`COPY` 就像是一条直通数据库仓库底层的传送带，而 `SELECT` 则像是一个需要经过层层审批和包装的快递员。

### 5.1.4. 导出对比

| 工具/命令        | 核心定位           | 输出格式               | 性能  | 权限要求           | 备注                             |
| :--------------- | :----------------- | :--------------------- | :---- | :----------------- | :------------------------------- |
| **`pg_dump`**    | 备份/迁移单个库    | SQL、压缩二进制、目录  | 中/高 | 普通用户(需SELECT) | 支持导出表结构+数据，最通用      |

## 5.2. 导入

### 5.2.1. copy

1. **与前一章相对应的，`COPY`是 PG 中性能最高的导入方式，亿级的数据导入效率远高于其它方式，不过导入前需要提前创建相应的表结构。**

它的导入语法结构和导出对比，主要就是从`TO`子句改为了`FROM`子句：

```sql
COPY { 表名 [ ( 列名 [, ...] ) ] | ( 查询语句 ) }
    FROM { '文件绝对路径' | PROGRAM '外部程序命令' | STDIN }
    [ WITH ( 选项 [, ...] ) ]
```

`COPY`导入依旧分为服务端和客户端两种方式：

- 服务端执行并导入到数据库：`COPY my_table FROM '/tmp/data.csv' WITH (FORMAT CSV, HEADER);`。这种方式要通过**数据库超管**执行 SQL 语句来完成，只能读取服务端的文件。
- 客户端执行并导入到数据库：`\copy my_table FROM 'C:/Users/Desktop/data.csv' WITH (FORMAT CSV, HEADER)`。需要用户在使用`psql`登录后再执行，可读取用户本地的文件。

------

2. **和导出不同，导入需要关心表结构，特别是索引、触发器这些会影响导入效率的“包袱”都需要注意。**

如果你直接往一个生产表的“全装状态”（带索引、带约束、带触发器）导数据，速度会慢得像蜗牛。**高效导入的逻辑是：先拆、后导、再装。**

**第一步**：卸掉“包袱”（索引与约束）

- **原理**：每插入一行，数据库都要实时更新索引、校验约束（如外键）。对于一亿条数据，这个开销是灾难性的。
- **操作**：先 `DROP INDEX` 掉该表的所有非必要索引，并禁用外键。导完后再重新 `CREATE INDEX`。**重建索引比实时维护索引快 10 倍以上。**

**第二步**：停掉“哨兵”（触发器）

- **操作**：`ALTER TABLE my_table DISABLE TRIGGER ALL;`
- **注意**：这要求你对数据的准确性有把握，因为触发器里的业务逻辑会被跳过。

**第三步**：批量提交与 WAL 优化

- **单事务提交**：将整个 `COPY` 放在一个事务中。
- **绕过日志 (WAL)**：如果你是向**新创建**的表中导数据，在同一个事务里创建表并 `COPY`，PG 会触发优化，不写 WAL 日志，速度快到飞起。

  ```sql
  BEGIN;
  CREATE TABLE t_new (...);
  COPY t_new FROM ...;
  COMMIT;
  ```

**第四步**：使用 `FREEZE` 参数

​        在导入时，`COPY`的`WITH`子句会多一个`FREEZE`参数可用，这个参数的作用是：

​        在数据入库时直接标记为“已冻结”，防止数据库之后再跑一次 `VACUUM` 来扫描这些新数据，极大减轻后续的系统维护压力。

**总结**：从上面的步骤可以看出来，`COPY`的导入快是牺牲数据校验得来的，如果数据文件的来源比较杂，那么花费在数据清洗上的时间可能并不一定比其他方式的导出后恢复花费时间短。

------

3. **导入时的“脏数据”**

产生脏数据的原因常见的有这几种：

- **约束不一致**：你导出的源库（A库）可能没有设置 `NOT NULL`、`CHECK` 约束或外键，但目标库（B库）的表结构定义更严格。
- **字符集与编码**：如果 A 库是 Latin1，B 库是 UTF-8，某些特殊字符导出来后再导进去就会报“无效的字节序列”。
- **版本差异**：高版本 PG 对数据类型的校验可能比低版本更严（例如某些非法的日期格式 `0000-00-00`）。
- **文件损坏/截断**：在大文件传输过程中，如果网络波动导致文件末尾缺失，或者 CSV 的引用符（Quote）没有成对出现，会导致 `COPY` 解析错位。
- **文件不规范**：获取的导入文件是人工创建或是由程序生成的，文件结构可能无法被 PG 正确解析。

如果导入时出现“脏数据”导致的异常，那么导入的整个事务都会回滚，为了尽可能避免这种情况出现，可以先进行预处理，通过`sed`或专门的程序进行检查校正。

如果没条件全量校验，那么分批导入能尽量避免数据全量回滚（在数据导出时就进行分批切割是最好的，如果是现成的文件那可以考虑对文件做行切割为多个文件）。

如果文件内容没问题，而是数据内容存在冲突，比如新表有唯一性约束而旧表没有，那么可以考虑先创建旧表结构，然后将数据导入旧表再通过`UPSERT`灌入新表。建表、导入和插入新表最好都在一个事务里执行，可以用到批量提交优化。

### 5.2.2. psql

`psql` 导入主要用于处理 `pg_dump` 导出的 **`.sql` 纯文本文件**。虽然它不支持并行功能，但它是唯一能“肉眼看懂”并手动修改的导入方式。

1. `psql` 导入的基本语法

在命令行界面（不是在 psql 内部），执行以下命令：

```bash
psql -U 用户名 -d 数据库名 -f /path/to/backup.sql
```

- **`-f`**: 指定要执行的 SQL 文件路径。
- **`-d`**: 目标数据库，可以用连接字符串，如果省略该参数会直接取用户名作为数据库名连接。
  
  > `psql "postgresql://用户名:密码@数据库地址:端口/数据库名" -1 --set ON_ERROR_STOP=on -f data_only.sql`

如果你之前用单行的 gzip 指令压缩了`pg_dump`导出的 SQL 文件，那么现在依旧可以用单行写法导入：

```bash
gunzip -c /path/data_only.sql.gz | psql "postgresql://用户名:密码@数据库地址:端口/数据库名" -1 --set ON_ERROR_STOP=on
```

如果你已经登录进了 `psql` 终端，可以使用内建指令：

```sql
\i /path/to/backup.sql
```

------

2. 为什么说 `psql` 导入很“单纯”？

`psql` 导入的本质就是：**把 SQL 文件里的内容，一行一行地发给数据库执行。**

- 如果文件里第一行是 `CREATE TABLE`，它就建表。
- 如果第二行是 `INSERT`，它就插数据。
- 它不具备 `pg_restore` 那种“解压”或“重排顺序”的能力，文件里怎么写的，它就怎么执行。

------

3. 两个必须掌握的“排坑”参数

在导入亿级数据生成的 SQL 脚本时，这两个参数能救命：

- **`-1` (或者 `--single-transaction`)**：
  - **作用**：将整个脚本包装在一个大事务中。
  - **优点**：万一导到一半报错，它会全部回滚，不会留下“半截”脏数据。而且在单事务中导入，速度会比一条条提交快得多。
- **`--set ON_ERROR_STOP=1`**：
  - **作用**：遇到错误立即停止。
  - **重要性**：默认情况下，`psql` 遇到报错会打印一行字，然后**继续执行**后面的语句。对于亿级数据，如果前面建表失败了，后面一亿条数据全都会报错刷屏，甚至导致不可预知的后果。

------

4. `psql` 导入的局限

- **单线程**：它没法像 `pg_restore -j` 那样多开。
- **文件体积**：如果是亿级数据，导出的 `.sql` 文件通常是巨大的（几十 GB 甚至上百 GB），光是读取这个文本文件并解析字符串，就会消耗大量的 CPU 和磁盘 IO。

### 5.2.3. pg_restore

如果说 `COPY` 是一次性搬运一整箱货，`psql`是简易的流水线，那么 `pg_restore` 就是一个**带导航和多工位的专业卸货系统**。

它专门用于处理 `pg_dump` 导出的 **自定义格式 (`-Fc`)** 或 **目录格式 (`-Fd`)** 备份文件。

------

1. `pg_restore` 的核心黑科技：并行还原 (`-j`)

这是在处理亿级数据时，它远强于 `psql` 的地方。

- **原理**：`psql` 还原 `.sql` 脚本是单线程的，按顺序一条条执行；而 `pg_restore -j [线程数]` 可以同时开启多个连接。
- **表现**：它能**同时恢复多个表**的数据。如果有 10 张大表，你开 `-j 8`，它就能同时往 8 张表里灌数据。
- **注意**：并行还原要求备份文件必须是 **`-Fc` (自定义格式)** 或 **`-Fd` (目录格式)**。

------

2. 常用还原场景命令

A. 最简还原（还原整个库）

```bash
pg_restore -U 用户名 -d 数据库名 backup.dump
```

B. 极致加速版（并行+报错忽略）

```bash
# 开启 4 个并行线程，提高还原速度
pg_restore -U 用户名 -d 数据库名 -j 4 backup.dump
```

C. “后悔药”：只还原某张表

如果你发现只有一张表数据坏了，不需要全库还原：

```bash
pg_restore -d 数据库名 -t my_important_table backup.dump
```

D. 只还原结构，不还原数据

```bash
pg_restore -d 数据库名 -s backup.dump
```

E. 使用连接字符串还原数据

```bash
pg_restore -d "postgresql://用户名:密码@数据库地址:端口/数据库名" -a -j 4 /path/data_only.dump
```

F. “边导边入”的骚操作

```bash
# 从生产库直接导向测试库，不产生中间文件
pg_dump -d "postgresql://user@prod_host/db" -Fc | pg_restore -d "postgresql://user@test_host/db"
```

------

3. `pg_restore` 的“智能”体现：目录列表 (`-l`)

你可以先不还原，而是把备份文件里的“目录清单”打印出来：

```bash
pg_restore -l backup.dump > toc.txt
```

在这个 `toc.txt` 里，你可以看到每一张表、每一个索引的恢复顺序。甚至你可以手动编辑这个文件，删掉不需要的部分，再通过 `-L` 参数让它按你修改后的清单还原。

------

4. 关键参数总结

- **`-d（--dbname）`**：数据库名，如果缺少该参数导入会变成打印，所有内容将会输出到屏幕上。
- **`-c（--clean）`**: 还原前先删除（Drop）现有的数据库对象，防止“对象已存在”报错。
- **`-v（--verbose）`**: 详细模式。它会告诉你现在正在还原哪张表、建哪个索引，亿级数据导入时能缓解你的焦虑。
- **`--no-owner`**: **非常实用。** 忽略原有的所有者信息。如果你把 `admin` 导出的数据还原到 `test_user` 下，这个参数能避免权限报错。
- **`-n <schema>`**: 只还原特定的模式。
- **`--disable-triggers`**：禁用触发器，可以提高导入效率。

| :--------------- | :----------------- | :-------------------- | :--- | :------------------------- | :--------------------------------------------------- |

## 5.3. 组合应用

1. 跨服务器的“无文件”克隆（SSH 隧道组合）

如果你有两台服务器 A 和 B，但 A 上没有足够的磁盘空间存临时文件，你可以通过 SSH 远程“喷射”数据：

- **组合方式**：`pg_dump` + `SSH` + `psql/pg_restore`
- **命令示例**：

  ```bash
  pg_dump -U user -d db_A -Fc | ssh remote_user@server_B "pg_restore -d db_B"
  ```

- **优点**：数据在内存中流动，不占用 A 服务器的任何磁盘空间，虽然因为加密传输会慢一点，但**安全且支持跨公网**，是目前外网传输最推荐的方案。

------

2. “一边导出、一边压缩、一边写入”（程序化组合）

如果你想在导出时就利用系统级的极速压缩（比如比 PG 自带压缩更猛的 `zstd` 或 `lz4`）：

- **组合方式**：`pg_dump` + `zstd` + `远程存储/命名管道`
- **命令示例**：

  ```bash
  pg_dump -d db_name -Fp | zstd -c > /path/to/db.sql.zst
  ```

- **还原时**：`zstdcat db.sql.zst | psql -d new_db`
- **优点**：针对亿级文本数据，`zstd` 的压缩率和速度通常优于 `pg_dump` 自带的 `-Fc`。

------

3. 命名管道组合（FIFO）

这是一种让不支持管道的工具也能“假装”走管道的高级技巧。

- **组合方式**：`mkfifo` + `pg_dump` + `你的处理程序`

- **操作逻辑**：
  1. 在系统创建个“假文件”：`mkfifo /tmp/pg_pipe`
  2. 让导出往这写：`pg_dump -d db -f /tmp/pg_pipe &`
  3. 让另一个程序从这读：`your_app /tmp/pg_pipe`
  
- **优点**：不需要真正的磁盘空间，但可以骗过那些只支持“读文件”而不支持“标准输入”的老旧工具。

  > **mkfifo** 的全称是 "Make First-In-First-Out"，属于 `coreutils` 包，**100% 所有的 Linux 发行版都原生自带**。它在 Linux 中创建的是一个**特殊的设备文件**。
  >
  > - **为什么能读取？**
  >
  >   对于程序（如 `pg_dump` 或你的代码）来说，它看起来就是一个普通的物理文件。程序调用标准的 `open()`、`read()`、`write()` 接口去操作它。
  >
  > - **读取的过程是什么样的？**
  >   这是一种**内存级别的中转**。
  >
  >   1. 当 `pg_dump` 向 FIFO 写入数据时，内核会将数据存入一个**内核缓冲区**（通常是 64KB），然后让 `pg_dump` 进程进入等待。
  >   2. 当读取程序（如 `psql`）打开这个文件读取时，内核直接把缓冲区里的数据交给它。
  >   3. **核心特征**：它必须**两端同时打开**。如果没有程序读，写的程序就会卡住；如果没有程序写，读的程序也会卡住。
  >
  > - **为什么不需要磁盘空间？**
  >   虽然你在文件夹里能看到这个文件（它有一个文件名），但它在磁盘上**只占用一个索引节点（Inode）**。实际的数据**永远不会落盘**，它只在操作系统的内存缓冲区里流动。一旦读完，内存就被释放。

------

4. COPY 与 Netcat (nc) 的“暴力”组合

如果你追求极致的传输速度，连数据库协议的封装都不想要了：

- **组合方式**：`COPY` + `nc`

- **操作逻辑**：
  - **接收端 (B)**：`nc -l 8888 | psql -c "COPY table FROM STDIN"`
  - **发送端 (A)**：`psql -c "COPY table TO STDOUT" | nc server_B 8888`
  
- **优点**：省去了复杂的加密握手和多余的封装，几乎是裸奔的数据传输，适合**受信任内网**环境下的物理极限提速。

  > **nc（Netcat）** 被誉为网络工具里的“瑞士军刀”。绝大多数发行版（Ubuntu, CentOS 等）都会预装，即使没装，一行指令就能装好（`yum install nc` 或 `apt install netcat`）。
  >
  > - **它是什么？**
  >   它是一个功能极其简单的网络工具，作用就是**在两个地址之间建立一个 TCP/UDP 的原始通道**。
  >
  > - **是纯网络协议传输吗？**
  >   是的。它不做任何协议封装，不关心传输的是 SQL 还是图片。它只是把 A 端的“标准输入”原封不动地通过网线推给 B 端的“标准输出”。
  >
  > - **为什么用它？**
  >   因为它比 SSH 快。SSH 会进行加密处理（消耗大量 CPU），而 `nc` 是裸奔的，在内网环境传输亿级数据时，`nc` 能跑满万兆网线。
  >
  > - **有什么风险？**
  >
  >   因为 nc 不保证包的完整性，所以有极小概率出现数据截断而不报错（如果 nc 没能正确传回退出状态码）。

------

5. 逻辑解密流（Logical Decoding）

这属于“准逻辑复制”的范畴。

- **组合方式**：`pg_recvlogical` + `自定义脚本` + `目标库`

- **原理**：它不是导出一份静态快照，而是持续导出**增量日志**。

- **场景**：当你需要把 A 库的变化实时“注入”到 B 库，又不想建立正式的复制关系时使用。

- **风险点**：位点（Slot）失效或 WAL 日志被覆盖。丢失增量数据，需要重做全量快照。

  > **逻辑复制槽 (Replication Slot)** 是数据库里的一个“记事本”，记录了某个订阅者（导出程序）读到了 WAL 日志的哪个位置（LSN）。
  >
  > - **WAL 日志被覆盖（The Overwrite Risk）**：
  >   - **原理**：PG 的日志文件（WAL）是循环使用的。如果导出程序（如 `pg_recvlogical`）挂了，或者网络断了很久，这个 Slot 就会停在原地。
  >   - **数据库的保护机制**：默认情况下，只要 Slot 还在，PG 就会强制保留这些 WAL 日志不被删除，**直到把你的磁盘撑爆**。
  >   - **失效点**：为了防止磁盘爆掉，DBA 通常会设置 `max_slot_wal_keep_size`（限制 Slot 能保留的最大日志量）。如果积压的日志超过这个值，PG 为了保命，会强制删掉旧日志。
  >   - **结果**：Slot 标记为 **invalid (失效)**。因为中间缺了一段日志，导出程序再连上来时发现“断档”了，它无法通过日志补齐数据，只能**删掉目标库数据，重新做一次全量快照导出**。

------

**核心总结：**

- **pg_dump + pg_restore**：官方标配，最稳。
- **管道 + SSH**：跨库迁移，省磁盘。
- **管道 + 压缩工具 (zstd)**：海量数据，省带宽和存储。
- **命名管道 (FIFO)**：适配特殊工具，玩花活。

**局限性：**

A. 脆弱性

- 不管是 `mkfifo` 还是 `nc`，它们都是**流式**的，无法实现断点续传。
- **局限**：如果传输以一条数据到 99% 时网络断了一下，或者某个进程崩了，整个过程会立即崩溃。你**无法从断点继续**，只能从头开始导这一亿条数据。

B. 错误捕获困难

- 在 `pg_dump | nc | pg_restore` 这种长链条中，如果中间的 `nc` 挂了，前端的 `pg_dump` 可能还在拼命导数据，导致错误信息被淹没在管道流中，排查起来非常头疼。

C. 安全风险

- `nc` 是**明文传输**。如果你在公网使用它搬运数据库，所有的敏感数据（如用户密码、身份证号）都是在互联网上裸奔的，极其危险。

D. 权限与阻塞

- `mkfifo` 要求两个进程必须在同一台机器上，或者通过复杂的挂载实现。如果读取端处理太慢，会导致写入端（数据库）的进程长时间处于“等待”状态，占用数据库连接资源。

E. 监控困难

- 管道中间的数据流是黑盒，你很难精确知道“还剩几分钟完成”，除非你去数目标表的行数。

F. 长事务导致 Read View 被占用

- **Read View 是什么？**
  它是一张“花名册”，记录了当前导出事务启动时，库里有哪些事务还没提交。导出事务只能看到“花名册”之外已经提交的数据。

  > **逻辑解密**与 **Read View** 的关系：
  >
  > 逻辑解密在启动初期（创建 Slot 时）也会短暂地占用 Read View 来确定起始位点。
  >
  > 但它比 `pg_dump` 聪明的地方在于：一旦进入增量同步阶段，它就不再持有长事务，而是通过**解析 WAL 日志**来获取数据。
  >
  > 这就避免了 `VACUUM` 被长期阻塞的问题，是亿级数据迁移中**唯一不严重影响在线业务性能**的方案。

- **为什么事务过长是万恶之源？（数据库膨胀）**：

  1. **垃圾不能回收**：PG 的 `VACUUM` 进程负责清理被删除或更新过的旧行（Dead Tuples）。
  2. **清理条件**：`VACUUM` 只能清理那些“所有活跃事务都看不见”的旧行。
  3. **阻塞点**：如果你的导出事务运行了 10 小时，那么这 10 小时内产生的所有旧数据（哪怕是被其他业务 `UPDATE` 或 `DELETE` 掉的），由于**你的 Read View 还在引用它们**，`VACUUM` 统统不敢删。
  4. **后果**：表体积疯狂增长（Table Bloat），索引失效，最终可能导致磁盘溢出或性能剧降。

------

## 5.4. 逻辑复制

**逻辑复制**是一种基于数据内容（逻辑对象）的实时流式同步技术。它不再搬运整个数据库文件，而是把数据库的变更（增量）像直播一样实时推送到另一台服务器上。

------

### 5.4.0. 版本支持与兼容性

逻辑复制是 PostgreSQL 近年来最重要的进化之一：

- **PG 10 之前**：原生不支持。必须引入第三方插件，最著名的是 **`pglogical`**。
- **PG 10 及以后**：正式引入**原生逻辑复制**，直接写 SQL 语句就能跑。
- **跨版本支持**：它最大的优势就是**支持跨大版本**（例如从 PG 11 同步到 PG 15）。这是 `pg_dump` 很难稳定做到的。
- **兼容性限制**：
  - 目前**不支持** DDL（建表、改字段名）自动同步。你必须手动在两边执行 `CREATE TABLE`。
  - 必须有**主键 (Primary Key)**。如果没有主键，更新和删除会非常慢甚至报错。

------

### 5.4.1. 三个核心组件

在上一章节的逻辑解密流中有提到**逻辑复制槽 (Replication Slot) **这个概念，在逻辑复制中与之相联系的还有**逻辑解密 (Logical Decoding) **和**发布订阅 (Pub/Sub) **。

- **逻辑解密 (Logical Decoding)**：这是“翻译官”。它负责把二进制的 WAL 日志“解码”成逻辑上的数据变更（比如：这条日志代表 INSERT 了一行）。
- **复制槽 (Replication Slot)**：这是“书签（类似 Kafka 的 Offset）”。它确保翻译官不会把进度弄丢，防止源库删掉订阅者还没读的日志。
- **发布订阅 (Pub/Sub)**：这是“通信协议”。定义了谁卖数据（发布），谁买数据（订阅），以及买哪些表。
  > 逻辑复制是通过 **发布（Publication）** 和 **订阅（Subscription）** 来实现的：
  >
  > - **发布 (Publication)**：在源库定义。你可以指定“我愿意分享哪些表的数据”。
  > - **订阅 (Subscription)**：在目标库定义。它连接到源库，并负责拉取和应用数据。

------

### 5.4.2. 它是如何工作的？（核心机制）

它的底层原理就像一个**“翻译+重放”**的过程：

1. **捕获**：源库在写入数据时，把变更记入 WAL 日志。
2. **解密**：后台进程将复杂的二进制日志“翻译”成人类可读的逻辑指令（例如：`表A 插入了一行 ID=1`）。
3. **传输**：通过网络将这些指令实时发给目标库。
4. **应用**：目标库收到指令后，在自己的表里执行同样的 `INSERT` 或 `UPDATE`。

------

### 5.4.3. 它与传统“物理复制”的区别

| 特性           | 物理复制 (Physical)      | 逻辑复制 (Logical)                     |
| :------------- | :----------------------- | :------------------------------------- |
| **同步对象**   | 整个数据库实例（全拷贝） | **指定特定的表**                       |
| **版本要求**   | 主从库版本必须严格一致   | **支持跨版本**（如 PG 11 到 16）       |
| **目标库状态** | 只能只读（Hot Standby）  | **可读可写**                           |
| **DDL同步**    | 自动同步所有操作         | **不支持 DDL**（改字段需手动在两边做） |
| **性能开销**   | 极低                     | 略高（解密和重放需要 CPU）             |

------

### 5.4.4. 它的“杀手锏”场景

为什么亿级数据迁移最后都要聊到它？

- **不停机迁移（Zero Downtime）**：先跑全量，再实时追增量。当两边数据同步时，只需几秒钟切换应用连接，业务几乎无感知。
- **跨平台/版本升级**：想从 CentOS 迁移到 Ubuntu，或者从 PG 12 升级到 16，逻辑复制是最佳工具。
- **数据聚合**：把 5 个分库的数据全都汇总到一个大仓库库里做分析。

------

### 5.4.5. 快速上手的三步走（SQL 命令）

**第一步：源库配置与发布**

修改配置文件 postgresql.conf 的日志等级（必须重启，让日志包含解密信息）。

```ini
# 必须项。默认是 replica，只有设为 logical，WAL 日志才会记录解密所需的逻辑信息。
wal_level = logical
# 允许创建的“书签”数量，按需设置。如果你有 3 个目标库要订阅，这个值至少要大于 3。
# 订阅达到上限后，新的订阅无法创建，报错“all replication slots are in use”。
max_replication_slots = 10
# 允许同时运行的同步连接数。通常设为与 Slot 数量一致或略大。
# 同步达到上限后，订阅者连不上源库，同步无法开始。
max_wal_senders = 10
# PG 13 起支持这个参数，默认 64MB，对于存在超大事务的场景调大后可以减少磁盘 IO
# 注意：总内存消耗 = max_replication_slots * logical_decoding_work_mem
# logical_decoding_work_mem = 64MB
```

向配置文件 pg_hba.conf 里添加订阅方的地址配置（不必重启，执行`SELECT pg_reload_conf();` 或者 `pg_ctl reload`即可）。

```ini
# 允许特定 IP 进行复制连接
host replication rep_user 192.168.1.100/32 md5
```

> 参数释义：
> 1. **`host`**: 匹配使用 TCP/IP 连接的客户端。
> 2. **`replication`**: 这是一个**特殊关键字**。表示这条规则是给复制连接（包括逻辑复制）用的，而不是普通数据库访问。
> 3. **`rep_user`**: 允许连接的用户名。
> 4. **`192.168.1.100/32`**: 目标库的 IP 地址。`/32` 表示精确匹配这一台。
> 5. **`md5`**: 认证方式。表示连接时需要输入加密后的密码。

通过 SQL 语句创建发布，包含你要同步的表

```sql
-- 创建发布规则，可以指定多个表，使用英文逗号分隔
-- 注意：创建这个规则不需要用户有 REPLICATION 权限
CREATE PUBLICATION 发布名 FOR TABLE 表1, 表2;
```

> 在 PG 15+ 起，每个表名后面都可以跟 where 条件子句过滤需要同步的数据行（**只能用本表列，不能有子查询也不能有复杂自定义函数**）。
>
> 对于已经发布的规则可以通过修改语句增加条件：
>
> ```sql
> ALTER PUBLICATION 发布名 SET TABLE 表1 WHERE (status = 'paid');
> ```
>
> **如果源库修改了数据行导致条件不满足，那么源库会发送一个`DELETE`操作给目标库删除数据，如果是从不满足到满足则会发送一个`INSERT`操作。**
>
> 如果源库发布了两条不同的规则包含同一个表但行过滤条件不同，并且目标库同时订阅了这两条规则，那么收到的数据会是两个过滤条件的并集。
>
> 例如，两个规则里 A 表条件分别为`WHERE id < 100`和`WHERE id > 50`，这两个条件只要满足其一就会被推送给目标库，从示例来看其实就是所有数据。

**第二步：目标库建立结构**

手动在目标库执行 `CREATE TABLE ...`，建好跟源库一样的表结构（**列名和类型要对齐，列可以多但不能少，但多出的列必须要可为空或携带默认值**）。

订阅规则里写的表和列在目标库里都要建，少一个表都不行（**列数据类型如果可兼容的话不必一致**，例如`int` -> `text`）。

如果表不一致在订阅时就会导致 PG 后台进程崩溃并不断重启（断流），日志中可看到大量报错`ERROR: relation "xxx" does not exist`。

> 多一列的应用：
>
> - 增加一列同步时间，默认值为当前时间，可用于监控数据的同步时间。
> - 增加一列业务数据，例如在高版本 PG 上使用生成列动态计算出一个新列给读写分离架构下的读库使用。

**第三步：目标库订阅**

```sql
-- 创建订阅规则，连接到源库并开始同步
CREATE SUBSCRIPTION 订阅名 
-- 和 pg_dump 一样可以使用两种结构的连接串
-- 注意：连接串里的用户必须有 REPLICATION 权限
CONNECTION 'host=源库IP port=5432 user=rep_user dbname=mydb' 
-- 指定要同步的订阅规则
PUBLICATION 发布名;
```

> 如果是列约束或数据冲突导致复制报错，那么订阅流程会崩溃退出并尝试重启（断流）。由于冲突一直在，所以重启也无济于事，后台进程会一直卡在这里。
>
> 1. 约束冲突场景：
>
> - **非空列冲突**：多的数据列为非空但未设置默认值。
> - **主键冲突**：源库插入 ID=1，目标库已经有一个 ID=1 了（重复键报错）。
> - **外键冲突**：外键表的数据还没同步导致子表数据同步时报错。
> - **数据缺失**：源库更新 ID=10 的行，但目标库找不到 ID=10（更新失败报错）。
> - **检查约束冲突**：源库允许 `age < 0`，但目标库设置了 `CHECK (age > 0)`。
>
> 2. 解决冲突的办法：
>
> - 对于非空约束冲突可以添加列的默认值或将列设为可空。
>
> - 对于主键冲突需要手动消除重复的主键。
>
> - 对于外键冲突一般等待外键表同步后就能自动消除，若外键表不同步则需要手动禁止外键触发器，等同步结束再恢复。
>
> - 对于数据确实冲突需要手动补足缺失的数据行。
>
> - 对于检查约束冲突需要手动调整目标库的约束设置来兼容冲突的数据。
>
> - 执行`pg_replication_origin_advance()`把同步位后移以跳过冲突位置（可能会造成数据不一致，慎用）。
>
> 3. 增加异常检测：
>
> - **在源库看`pg_stat_replication`**：关注 `sent_lsn` 和 `write_lsn` 的差距。如果 `reply_lsn` 停滞不前，说明订阅者可能挂了。
> - **在目标库看`pg_stat_subscription`**：这是最重要的！查看 `last_msg_receipt_time`（最后接收时间）。如果这个时间很久没更新，说明流断了。
> - **看目标库日志**：订阅进程报错后，会疯狂往 PostgreSQL 的日志文件（`pg_log`）里刷报错信息（比如 `duplicate key value...`）。**监控日志里的关键字是最直接的方法。**

第四步：增加同步表

```sql
-- 在源库添加新表到发布规则里
ALTER PUBLICATION 发布名 ADD TABLE 新同步的表名;
```

第五步：刷新订阅规则

```sql
-- 注意：结尾不需要再声明一次发布规则的名称
ALTER SUBSCRIPTION 订阅名 REFRESH PUBLICATION;
```



------

### 5.4.6. 潜在的问题

- 如果有 `UPDATE` 或 `DELETE` 操作，表必须有主键。如果没有主键，你必须在源库设置 `REPLICA IDENTITY FULL`，但这会导致每一行更新都要全表扫描寻找匹配项，对于大数据量同步来说性能会直接崩盘。

- 如果发生 WAL 被删，Slot 失效的情况，逻辑复制会停摆，为了尽量避免这种情况的出现，有两套应对方案：

  - **官方方案（重刷）**：目前原生逻辑复制**无法自动补齐**丢失的中间日志。一旦失效，订阅端通常会报错停止。你必须重新执行全量同步（`ALTER SUBSCRIPTION ... REFRESH PUBLICATION`），但这在亿级数据下很痛苦。

  - **预防方案（DBA 常用）**：
    - **设置保留上限**：`max_slot_wal_keep_size` 设大一点，给网络波动留出几天甚至几周的容错时间。
    - **监控报警**：实时监控 Slot 的延迟，在日志被删前人工干预（比如修复网络或扩容磁盘）。


------

### 5.4.7. 💡 总结

逻辑复制就是一种**“精细化、实时化”**的搬运方式。它解决了 `pg_dump` 停机时间长的问题，也解决了物理复制“必须全库拷贝、版本必须一致”的死板问题。

由于它的实现核心是**发布/订阅**模式，因此存在一定的延迟，对于高可靠性与高一致性有要求的主从数据库等场景不适用，对于读写分离、最终一致的场景可考虑。

# X. 如何基于源码在 Linux 上部署 PG

官方安装文档地址：https://www.postgresql.org/docs/current/installation.html

源码包下载地址：https://www.postgresql.org/ftp/source/

1. 先从https://www.postgresql.org/ftp/source/下载所需版本的源码（bz2的包）

2. 使用`tar xf postgresql-<version>.tar.bz2`解压源码包并进入其中

3. 使用`./configure`检查编译环境，如果有不满足的会提升，可以试着根据提示加参数屏蔽某些检查

4. 直接执行`make`开始编译，正常来说执行完就编译完了，如果失败了就根据错误信息进行搜索

5. 使用`su`切换到 root 权限，再执行`make install`开始 PG 的安装

6. 添加 postgres 用户：`adduser postgres`并设置密码

7. 为 PG 创建数据目录，可以建在 PG 的安装目录（`/usr/local/pgsql`）也可以另外找目录

   ```sh
   mkdir -p /usr/local/pgsql/data
   或
   mkdir -p /opt/pg-data
   ```

8. 为数据目录设置所属`chown postgres /opt/pg-data`

9. 切换到 postgres 用户`su - postgres`

10. 初始化数据目录`/usr/local/pgsql/bin/initdb -D /opt/pg-data`

11. 启动 PG 服务`/usr/local/pgsql/bin/pg_ctl -D /opt/pg-data -l logfile start`

12. 【可选】超管密码可能需要在 pg_hba.conf 里启用受信地址后登录上去指定

    ```sh
    /usr/local/pgsql/bin/psql postgres
    ALTER USER user_name WITH PASSWORD 'new_password';
    ```

13. 【可选】远程登录需要修改 postgresql.conf 启用 listen_addresses 配置 ( * 表示所有)

14. 【可选】创建数据库`/usr/local/pgsql/bin/createdb test`

15. 【可选】连接数据库`/usr/local/pgsql/bin/psql test`
